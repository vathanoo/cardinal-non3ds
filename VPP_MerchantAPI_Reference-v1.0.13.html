Overview

Visa Payment Passkey (VPP) is a FIDO-based authentication solution designed to help enhance security and user experience for e-commerce payments. It enables device-bound authentication across all Visa card-not-present (CNP) acceptance solutions with a single registration, allowing consumers to authenticate payments using biometrics or other secure methods without relying on passwords or OTPs.

This is a formal specification of VPP REST APIs for merchant integration.

Prerequisites

To access VPP APIs, developers need to register with, and then log in to, Visa Developer Center (VDC) or VPPS Product to obtain required credentials in the form of API Keys and other cryptographic key material.

In the sandbox, VPP APIs can be used by any developer with registration and acceptance of VDP terms and conditions; however, only participating Visa partners will be able to use these APIs in production.

In the event of any conflict between this specification and the VDP Agreement, the VDP Agreement shall govern and control.

General Notes and Conventions Used

  • The Visa-generated identifiers should be considered opaque references. Except for length, no other assumptions should be made.
  • Fields requiring encryption are prefixed with "enc", for example, encAccountNumber.
  • All resource and field names are in lowerCamelCase - a variation of Camel Case in which the first letter of the word is lowercase.
  • All strings must use in UTF-8 characters
  • All Date-Time fields use the ISO 8601 Date and Time Format of YYYY-MM-DDThh:mm:ssZ in UTC time or with the timezone suffix.
  • All Amount fields use a representation in minor units only. Note that there are no fractional minor units represented. Numerical value in the minor units of the currency - Cents in USD. For example, "12550" is used for representing the amount, "125.50".

The X-VIA-HINT header is used to provide a hint to the Visa network about the location of the client. This hint is used to optimize the routing of the request to the nearest Visa data center. The value of the X-VIA-HINT header is a two-letter country code (ISO 3166-1 alpha-2) that represents the country where the client is located. The X-VIA-HINT header is optional. If the header is not provided, the Visa network will use other methods to determine the location of the client.

prompt
required
string (Prompt) <= 25 characters
Enum: "login" "create" "none" "consent" "select_account"

Specifies how the end-user must be involved during the authentication process. Its value is a space-delimited string. Possible values include:

  • 'login': The end-user must be actively authenticated.
  • 'create`: The end-user is to be enrolled.
  • 'none': The end-user is authenticated silently, without their active participation (not implemented).
  • 'consent': The end-user must consent before returning any information to the Client (not implemented).
  • 'select_account': The end-user might select an account if multiple sessions with different accounts are already available (not implemented).
amr_values
Array of strings (AMR_VALUES)

The authentication methods being requested. Note: For VPPS only pop#fido2 is used. Values from OIDC:

  • pop: Proof of Possession. Similar to swk and hwk without specifying if it is through software or hardware. Custom Values:
  • fed: Federated. These values might be qualified for more granularity: Qualifier Values:
  • When amr is 'otp':
    • otp#time: Time-based OTP.
    • otp#hash: Hash-based OTP.
    • otp#email: OTP via email.
    • otp#sms: OTP via SMS.
    • otp#txn: OTP with a payment transaction amount.
  • When amr is 'pin':
    • pin#cvv: PIN with CVV.
  • When amr is 'tel':
    • tel#csr: Telephone with customer service representative.
    • tel#cb: Telephone callback.
  • When amr is 'rba':
    • rba#rf: Risk-Based Authentication with refresh token.
    • rba#aref: Risk-Based Authentication with activity reference.
    • rba#sidt: Risk-Based Authentication with secured ID token.
    • rba#csrt: Risk-Based Authentication with cross-site recognition token.
    • rba#tpt: Risk-Based Authentication with third-party token.
  • When amr is 'pop':
    • pop#fido2: Proof of Possession with FIDO2.
  • When amr is 'fed':
    • fed#oidc: Federated with OpenID Connect protocol.
    • fed#tds: Federated with 3DS protocol.
    • fed#saml2: Federated with SAML2 protocol.
    • fed#vtsiss: Federated with VTS IdV specification.

  • S256: Calculated as BASE64URL-ENCODE(SHA256(ASCII(code_verifier))).
